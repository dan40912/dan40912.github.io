# AGENTS.md — 個人 AI 學習知識庫操作規範

這個 Obsidian vault 是使用者的長期 AI 學習記憶庫。回答 AI、Agent、MCP、Skill、RAG、Prompt、LLM、Codex、Claude、工作流與自動化相關問題時，必須優先讀取本 vault，而不是只靠模型記憶。

## Vault 根目錄

真正的 vault 根目錄是：

`D:\obsidian\個人學習\個人學習\`

## 回答前的檢索順序

1. 先讀 `00_首頁 Home/🏠 知識庫首頁.md` 與 `02_關係地圖 Maps/🗺️ AI 全景地圖.md`，確認目前知識架構。
2. 依問題主題讀 `01_核心概念 Concepts/` 的對應筆記。
3. 若問題涉及做法、流程或工具選型，再讀 `03_應用與工作流 Applications/`。
4. 若問題涉及最新 AI 動態，再查 `04_每日快訊 Daily/`、`05_每週精選 Weekly/`；必要時再上網核對。
5. 回答時盡量用 `[[筆記名]]` 指回本 vault 的相關筆記。

## 答案風格

- 主要用繁體中文回答，專有名詞保留英文並補中文說明。
- 優先把抽象概念落成「我可以怎麼用」。
- 當 vault 內容不足、過時或互相矛盾時，要明確指出缺口，並建議新增或更新哪一篇筆記。
- 對重要知識問題，先說「根據 vault 目前脈絡」，再補充外部最新資訊。

## 寫入與更新規則

- 新概念放入 `01_核心概念 Concepts/`，沿用 `99_範本 Templates/概念筆記範本.md`。
- 新工作流、案例、專案規劃放入 `03_應用與工作流 Applications/`。
- 臨時想法先放 `06_收件匣 Inbox/`，之後再整理。
- 每篇筆記都要有 frontmatter：`type`、`title`、`tags`、`status`、`level`、`updated`、`related`。
- 新筆記至少連回 2-3 篇既有筆記，避免成為孤島。

## Codex / Claude 協作原則

- Codex 適合：讀寫本機檔案、整理 vault、建立規則、執行腳本、檢查結構、把想法落成文件。
- Claude 適合：長文整合、辯論式審查、語氣修整、跨文件脈絡總結。
- 多 Agent 適合：大型研究、專案規劃、知識庫稽核、重要決策審查；簡單問答不要硬上多 Agent。
- 多 Agent 任務至少分成：研究者、規劃者、審查者、整合者。整合者必須列出分歧點、採納理由與未採納理由。

## 每次維護 vault 的完成標準

- 有明確新增或更新的筆記。
- 有連結到首頁、地圖或相關概念。
- 有列出下一步。
- 若使用外部資料，有保留來源連結與日期。

## 與 Claude 協作（Handoff Protocol）

Codex 在此扮演**實作者 Builder + 整理者 Archivist**；Claude 扮演**規劃者 + 審查者**。交接靠 `_協作 Collab/` 的固定檔案，不靠對話記憶。

- **開工前**：讀 `_協作 Collab/task.md`（任務與驗收標準）。若由 `orchestrate-vault.ps1` 啟動，task.md 已自動填好。
- **執行**：依本檔的 vault 操作規範實作；只做 task 範圍內的事，超出範圍的列入開放問題。
- **收工後**：把「做了什麼 / 改了哪些檔 / 假設 / 開放問題 / 自我檢查」寫進 `_協作 Collab/report.md`，交給 Claude 審查。
- **整合**：收到 `_協作 Collab/review.md` 後，判斷採納哪些、直接修正，並在 report.md 列採納與未採納理由。
- 角色提示範本在 `_協作 Collab/prompts/`（01 規劃→02 實作→03 審查→04 整合）。
