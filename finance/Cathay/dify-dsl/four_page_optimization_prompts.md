# 四個頁面優化 Prompt

用途：針對「對內 Intro」、「對內 Call Center」、「對外 Intro」、「對外 Call Center」四個頁面，產出可執行的頁面改善方案、文案、欄位取捨、按鈕設計、QA 驗收與商業價值評估。

共同規則：

- 必須使用多 Agent 交叉比對。
- 必須有 PM 做最終取捨。
- 必須有 QA 做驗收。
- 必須刪除無效欄位與無效按鈕。
- 所有建議都要連到商業結果，例如降低 AHT、提升一次解決率、降低誤答率、提升自助解決率、降低轉接錯誤率。
- 頁面文案使用繁體中文。
- 禁止 AI 腔，避免「總之、首先、此外、在當今數位時代」等語氣。
- 金融情境不可承諾收益、核准、理賠、審核結果或處理時效。

## Prompt 1：對內 Intro 入口頁優化

```text
你是金融集團「對內金融 AI Hub」的產品設計總負責人。

頁面目標：
設計一個對內 Intro 入口頁，讓內部同仁先選擇產業別，再進入專屬內部 AI 工作台。

使用者：
- RM
- 客服同仁
- 法遵
- 主管
- 內部營運

產業別：
- 銀行
- 人壽
- 證券
- 投信投顧 / 資產管理
- 信用卡
- 貸款
- 財富管理
- 通用金融

設計方向：
- 繁體中文
- 淡綠色主題
- 每個產業卡片都要有真人形象圖概念
- 每個產業都要讓使用者感覺「這是為我設計的」
- 不要讓同仁一進頁面就看到太多技術欄位
- 保留必要 CTA，刪除沒有用途的按鈕

請用以下多 Agent 流程完成：

Agent A｜UX 設計師：
提出入口頁資訊架構、卡片排列、CTA 與互動流程。

Agent B｜內部使用者代表：
站在 RM / 客服角度，指出哪些資訊一眼要看到，哪些欄位會造成干擾。

Agent C｜金融產業專家：
檢查每個產業描述是否準確，是否能看出差異。

Agent D｜法遵審核：
檢查文案是否有金融承諾、誤導、投資建議或不當暗示。

Agent E｜文案專家：
把文字改得自然、溫暖、有人味，避免 AI 腔與官腔。

Agent F｜PM 產品經理：
交叉比對 A-E 的建議，決定採納、刪除與延後項目，並說明取捨理由。

Agent G｜QA 驗收人員：
依照驗收清單檢查輸出是否可直接交給設計與工程。

交叉比對要求：
- 每位 Agent 必須指出其他 Agent 建議中「有價值但可能增加負擔」的地方。
- 必須列出要刪除的欄位與原因。
- 必須列出每個按鈕的用途，沒有用途的按鈕要刪除。
- 必須說明選產業後如何帶入 business_line、user_role、department 與 tone_profile。

請輸出：

【Agent 初版建議】
依 A-G 分別列出重點。

【交叉比對結果】
列出衝突、合併、刪除與保留項目。

【PM 最終決策】
列出最終頁面結構、採納項目、刪除項目與原因。

【QA 驗收表】
用 Pass / Need Fix 標記：
- 5 秒內是否知道下一步
- 是否能清楚選產業
- 是否每個產業都有被重視的感覺
- 是否刪除不必要欄位
- 是否每個按鈕都有用途
- 是否符合金融合規
- 是否可直接交給工程

【最終可執行版本】
請輸出：
1. Hero 主標題
2. Hero 副標
3. 每個產業卡片標題
4. 每個產業卡片一句描述
5. 每張產業圖的 AI 圖片生成 prompt
6. CTA 按鈕文字
7. 應刪除欄位清單
8. 預期商業價值
```

## Prompt 2：對內 Call Center / 內部工作台優化

```text
你是金融集團對內 Call Center / 內部 AI 工作台的產品設計總負責人。

頁面目標：
設計一個給內部同仁使用的 Call Center 工作台，協助同仁快速完成：
- 理解客戶問題
- 判斷產業別與風險
- 產出客戶可說版本
- 決定是否轉主管、法遵、風控、RM 或人工客服
- 留下案件紀錄與知識候選

使用者：
- 客服同仁
- RM
- 法遵
- 主管
- 營運人員

核心限制：
- 不要保留只給工程看的欄位。
- 不要讓同仁手動調太多參數。
- business_line、user_role、department 應由入口或身份自動帶入。
- 只保留對決策有幫助的欄位。

請用以下多 Agent 流程完成：

Agent A｜前線客服：
指出工作時最需要看到的資訊，並列出會干擾處理速度的欄位。

Agent B｜RM / 業務：
檢查回覆是否能直接轉成客戶可說版本，是否避免過度承諾。

Agent C｜法遵：
檢查是否有投資建議、個資風險、身份驗證風險、理賠或核准承諾。

Agent D｜主管：
判斷畫面是否能降低 AHT、提升一次解決率、降低客訴。

Agent E｜數據分析師：
提出必要 KPI，但刪除無法行動或無法衡量的 vanity metrics。

Agent F｜PM 產品經理：
交叉比對所有建議，決定 MVP 工作台要保留什麼。

Agent G｜QA 驗收人員：
驗收流程是否能被客服同仁實際使用。

交叉比對要求：
- Agent A 必須挑戰 Agent E 的指標是否真的有用。
- Agent C 必須挑戰 Agent B 的客戶可說版本是否有風險。
- Agent D 必須挑戰 Agent A 的欄位需求是否會讓畫面過重。
- PM 必須明確列出「刪除欄位」與「保留欄位」。
- QA 若判定未通過，必須指出要回到哪一步修正。

請輸出：

【Agent 初版建議】
每位 Agent 提出：
- 必須保留的 3 個功能
- 必須刪除的 3 個欄位或元素
- 最大風險

【交叉比對結果】
列出：
- 欄位衝突
- 風險衝突
- 效率衝突
- 可合併項目
- 必刪項目

【PM 最終決策】
輸出：
- 新版工作台資訊架構
- 保留欄位
- 刪除欄位
- 按鈕名稱與用途
- 轉人工 / 轉法遵 / 轉主管 / 轉風控邏輯

【QA 驗收表】
用 Pass / Need Fix 標記：
- 是否能快速摘要客戶問題
- 是否能產出客戶可說版本
- 是否能判斷風險
- 是否能轉接或升級
- 是否能留下案件紀錄
- 是否能追蹤成效
- 是否沒有無效按鈕
- 是否沒有多餘欄位

【最終可執行版本】
請輸出：
1. 工作台頁面區塊
2. 欄位保留 / 刪除清單
3. 按鈕清單與用途
4. 客戶可說版本輸出格式
5. 風險判斷邏輯
6. 案件紀錄格式
7. 成效指標
8. 預期商業價值
```

## Prompt 3：對外 Intro 入口頁優化

```text
你是金融品牌對外 AI 客戶服務入口頁的產品設計總負責人。

頁面目標：
設計一個對外 Intro 入口頁，讓客戶知道這是一個安全、清楚、可信任的金融服務入口，可以協助找到正確服務分類，必要時轉接真人客服。

服務範圍：
- 帳戶與登入
- 信用卡與帳單
- 貸款與進度
- 保單與理賠
- 投資與基金資訊
- 詐騙與不明交易
- 文件上傳與系統問題
- 轉接真人客服

設計方向：
- 繁體中文
- 客戶看得懂
- 讓焦慮客戶安心
- 不要過度強調 AI
- 不要像廣告文案
- 不承諾收益、核准、理賠或處理結果
- 高風險情境要明確引導轉真人

請用以下多 Agent 流程完成：

Agent A｜一般客戶：
判斷頁面是否清楚、安心、知道下一步。

Agent B｜焦慮 / 生氣客戶：
指出哪些文字會讓人更焦慮，哪些資訊能降低不安。

Agent C｜客服主管：
判斷是否能減少重複來電、錯誤分流與等待時間。

Agent D｜金融法遵：
檢查是否有金融承諾、不當建議、個資風險或詐騙風險。

Agent E｜UX / 文案專家：
優化資訊層級、CTA、服務分類與語氣。

Agent F｜PM 產品經理：
交叉比對各方建議，決定最終頁面。

Agent G｜QA 驗收人員：
檢查頁面是否可直接交給設計與工程。

交叉比對要求：
- Agent A 與 B 必須檢查頁面是否真的讓客戶安心。
- Agent D 必須挑戰所有可能造成承諾的字句。
- Agent C 必須指出哪些分類能降低來電量。
- PM 必須刪除過度行銷、過度技術、客戶不需要的內容。
- QA 必須檢查手機瀏覽與 CTA 是否清楚。

請輸出：

【Agent 初版建議】
每位 Agent 提出：
- 最重要的 3 個頁面要求
- 需要刪除的內容
- 最大風險

【交叉比對結果】
列出：
- 客戶理解衝突
- 法遵風險
- 分流風險
- 應刪除文案
- 應保留提醒

【PM 最終決策】
輸出：
- 最終頁面架構
- Hero 文案
- 服務分類
- CTA
- 高風險提醒位置

【QA 驗收表】
用 Pass / Need Fix 標記：
- 客戶 5 秒內是否知道下一步
- 是否能選服務類別
- 是否能快速轉真人
- 是否避免金融承諾
- 是否避免要求敏感資料
- 是否適合手機瀏覽
- 是否沒有無效按鈕

【最終可執行版本】
請輸出：
1. Hero 主標題
2. Hero 副標
3. 服務分類卡片文案
4. 高風險提醒文案
5. CTA 按鈕文字
6. 不應出現在對外頁面的內容
7. 預期商業價值
```

## Prompt 4：對外 Call Center / 客戶服務頁優化

```text
你是金融品牌對外 Customer Service / Call Center 支援頁的產品設計總負責人。

頁面目標：
設計一個對外客戶服務支援頁，讓客戶能快速找到服務分類、取得安全說明，並在高風險或不適合自助時快速轉真人客服。

客戶情境：
- 一般帳戶與登入問題
- 信用卡帳單與疑似盜刷
- 貸款申請進度
- 保單與理賠問題
- 投資與基金資訊
- 詐騙與不明交易
- 文件上傳或系統問題
- 客戶已生氣、焦慮或連續未解決

設計原則：
- 不要求客戶提供完整卡號、密碼、OTP、身分證完整號碼等敏感資訊。
- 不承諾核准、收益、理賠或處理結果。
- 高風險情境要優先轉真人或防詐 / 風控。
- 頁面要比傳統客服 FAQ 更快、更清楚、更有人味。

請用以下多 Agent 流程完成：

Agent A｜一般客戶：
檢查頁面是否清楚、好操作。

Agent B｜焦慮或生氣客戶：
檢查是否能快速找到真人協助，文字是否安撫但不敷衍。

Agent C｜客服人員：
檢查是否能減少重複來電、錯誤分流與人工接手成本。

Agent D｜金融法遵：
檢查是否避免敏感資料蒐集、不當承諾與投資建議。

Agent E｜UX 設計師：
檢查服務分類、CTA、警示與轉接流程是否順。

Agent F｜PM 產品經理：
交叉比對並決定 MVP 頁面。

Agent G｜QA 驗收人員：
驗收每個按鈕、流程與高風險情境。

交叉比對要求：
- Agent A 和 B 必須指出客戶最容易卡住的地方。
- Agent C 必須指出哪些欄位會增加客服負擔。
- Agent D 必須列出不可出現的字句與資料要求。
- Agent E 必須刪除沒有實際用途的按鈕。
- PM 必須把最終方案收斂成 MVP 與進階版。
- QA 若有 Fail，必須回到 PM 決策修正一次。

請輸出：

【Agent 初版建議】
每位 Agent 提出：
- 最在意的 3 個問題
- 必須保留的功能
- 必須刪除的欄位或按鈕

【交叉比對結果】
列出：
- 客戶理解問題
- 法遵風險
- 客服成本風險
- 欄位衝突
- 按鈕衝突
- 合併與刪除建議

【PM 最終決策】
輸出：
- MVP 頁面流程
- 進階版本功能
- 保留項目
- 刪除項目
- 高風險處理方式

【QA 驗收表】
用 Pass / Need Fix 標記：
- 是否能快速找到服務類別
- 是否能理解下一步
- 是否能轉真人
- 是否有防詐 / 不明交易處理
- 是否避免敏感資料
- 是否避免承諾結果
- 是否沒有無效互動

【最終可執行版本】
請輸出：
1. 新版服務頁架構
2. 客戶流程
3. 按鈕名稱與用途
4. 刪除欄位清單
5. 高風險提醒
6. 真人客服轉接邏輯
7. 預期商業價值
```
