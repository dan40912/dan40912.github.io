# 四個頁面產出方案

本文件依照「多 Agent 交叉比對、PM 決策、QA 驗收」流程，直接產出四個頁面的可執行方案：

1. 對內 Intro 入口頁
2. 對內 Call Center / 內部工作台
3. 對外 Intro 入口頁
4. 對外 Call Center / 客戶服務頁

---

## 1. 對內 Intro 入口頁

### Agent 初版建議

UX 設計師：
- 首屏只做一件事：讓同仁選產業。
- 產業卡片要大，每張卡要有人物情境圖。
- CTA 用「進入工作台」，不要用模糊的「了解更多」。

內部使用者代表：
- RM / 客服不想先看技術參數。
- 入口要直接講這個助理能幫他處理什麼。
- 每個產業要看起來不同，不要像同一張卡換標題。

金融產業專家：
- 證券、人壽、銀行、信用卡的風險邊界差異要明確。
- 通用金融要定位為「先追問、再分流」。
- 財富管理不可看起來像投資推薦頁。

法遵審核：
- 不可用「保證」、「推薦」、「最佳投資」、「快速核准」。
- 對內入口可提效率，但不可暗示 AI 可取代審核。
- 高風險業別需保留適合度、身分驗證與風險揭露提醒。

文案專家：
- 主標要有人味：「每個產業，都應該被認真對待」。
- 副標要說清楚為什麼先選產業。
- 卡片文案要像對同仁說話，不像系統設定。

PM：
- MVP 只保留入口 hero、產業卡片、三個價值說明。
- 不放全域搜尋、不放手動參數、不放工程設定。
- 產業選取後再進入工作台，不要讓使用者先看到 console。

QA：
- 首屏需在 5 秒內知道下一步。
- 每張產業卡都要能點。
- 所有按鈕必須有明確功能。

### 交叉比對結果

保留：
- Hero 主標與副標
- 8 張產業卡
- 真人情境圖
- 進入工作台 CTA
- SSO / 內部權限 / Dify 已串接狀態

刪除：
- 全域搜尋：入口階段不需要，會分散注意力。
- 手動角色切換：角色應由產業與登入身份帶入。
- 技術 payload：入口頁不應給一般同仁看。
- 多餘說明卡：超過三個價值主張就會降低掃讀效率。

衝突：
- 數據代理人想放更多 KPI，但 UX 與 PM 判定入口不應承載管理報表。
- 產業專家想放更細 SOP，但法遵與 PM 判定應進工作台後再顯示。

### PM 最終決策

頁面結構：
1. Topbar：品牌、SSO、內部權限、Dify 狀態
2. Hero：主標、副標、快速進入證券工作台、查看所有產業
3. 產業選擇：8 張大卡，每張有人物圖、產業描述、tags、CTA
4. 底部價值：給前線同仁、給法遵管理者、給營運團隊

### 最終可執行版本

Hero 主標：
「每個產業，都應該被認真對待。」

Hero 副標：
「同仁不應該先進入一個通用聊天機器人。這個入口會先判斷產業別，再載入正確的規則、角色、部門、風險提醒與回覆流程，讓阿發像是為該業別準備的內部助理。」

CTA：
- 進入證券工作台
- 查看所有產業
- 開啟銀行服務助理
- 開啟人壽保險助理
- 開啟證券服務助理
- 開啟投信投顧助理
- 開啟信用卡服務助理
- 開啟貸款服務助理
- 開啟財富管理助理
- 開啟通用金融分流助理

產業卡片文案：
- 銀行服務助理：帳戶、開戶、登入、轉帳、存款、貸款、信用卡、詐騙與不明資金。
- 人壽保險助理：保單保障、理賠、保單價值、保單借款與減額繳清。
- 證券服務助理：開戶、下單、交割、庫存、複委託、融資融券與風險揭露。
- 投信投顧助理：基金、淨值、申購、贖回、配息、產品文件與風險等級。
- 信用卡服務助理：申辦、額度、帳單、繳款、爭議款、疑似盜刷、停卡與補發。
- 貸款服務助理：房貸、信貸、車貸、利率說明、補件與申請進度。
- 財富管理助理：產品資訊、風險屬性、資產配置說明與預約專員。
- 通用金融分流助理：業別不明或跨業別問題，先追問再導入正確規則。

圖片生成方向：
- 銀行：嚴謹 RM 看帳戶文件與平板。
- 人壽：顧問拿電話關心客戶並做筆記。
- 證券：證券 RM 看 K 線與交易螢幕，表情專注。
- 投信投顧：基金專員看配置圖與風險文件。
- 信用卡：客服專員戴耳機看疑似盜刷提醒。
- 貸款：貸款專員審核房貸文件。
- 財富管理：財管顧問在諮詢室討論長期規劃。
- 通用金融：內部營運人員看多流程分流畫面。

### QA 驗收

| 項目 | 結果 |
|---|---|
| 5 秒內知道下一步 | Pass |
| 能清楚選產業 | Pass |
| 每個產業有不同形象 | Pass |
| 無多餘技術欄位 | Pass |
| 每個按鈕有用途 | Pass |
| 符合金融合規 | Pass |

### 商業價值

- 降低錯選業別造成的錯誤回答。
- 減少同仁理解系統的時間。
- 提升進入 chatbot 前的分流正確率，目標達 95% 以上。

## 2. 對內 Call Center / 內部工作台

### Agent 初版建議

前線客服：
- 需要快速看到客戶問題、風險、建議回覆。
- 不需要看到太多模型參數。
- 需要一鍵轉主管、法遵、風控。

RM / 業務：
- 回覆要能直接改成客戶可說版本。
- 投資、貸款、保險不能承諾結果。
- 要能保留「內部提醒」。

法遵：
- 必須保留禁答邊界。
- 高風險情境要有升級路徑。
- 所有回答要能留下稽核紀錄。

主管：
- 要看案件量、等待時間、一次解決率、風險攔截。
- 不要讓頁面變成報表系統。
- 按鈕要能驅動下一步。

數據分析師：
- 保留 AHT、重寫率、知識重用率、風險攔截。
- 刪除無法行動的 vanity metrics。

PM：
- MVP 保留：總覽、案件、合規、知識、成效。
- 移除手動角色切換與全域搜尋。
- 右側固定 rail 放目前助理、需求、Dify、payload。

QA：
- 案件點擊後需載入需求。
- 按鈕要有 toast 或狀態變化。
- payload 要跟業別同步。

### 交叉比對結果

保留欄位：
- 目前助理
- 禁答與風險提醒
- 同仁需求
- 開啟 Chatbot
- 複製參數
- 技術 inputs 預覽
- 案件編號、主題、業別、風險、狀態、等待、需求

刪除欄位：
- 全域搜尋
- 手動 staff role
- 手動 department
- 無行動價值的圖表
- 假通知按鈕
- 無功能匯出按鈕

保留按鈕：
- 開啟 Chatbot
- 複製參數
- 重新選產業
- 收合 Chatbot
- 標記已審核
- 升級所選案件
- 通過項目
- 要求修正
- 儲存知識候選
- 複製 KPI 摘要

### PM 最終決策

工作台頁面：
1. 工作台總覽
2. 案件佇列
3. 合規審核
4. 知識營運
5. 成效分析

右側固定欄：
1. 目前助理
2. 同仁需求
3. Dify Chatbot
4. 技術參數

### 客戶可說版本輸出格式

```text
客戶可說版本：
（給同仁可直接複製給客戶的文字）

內部提醒：
（不可承諾、需確認、需升級的事項）

需確認資料：
（身分、商品、日期、文件、案件狀態）

是否需轉接：
不需 / 主管 / 法遵 / 風控 / RM / 技術
```

## 3. 對外 Intro 入口頁

### Agent 初版建議

一般客戶：
- 要知道可以問什麼。
- 要知道什麼時候會轉真人。
- 不想看到太多內部名詞。

焦慮客戶：
- 需要看到「疑似詐騙、盜刷、不明交易」入口。
- 需要立即找到真人客服。
- 文案要安撫，但不要敷衍。

客服主管：
- 分類要能降低錯誤來電。
- FAQ 不要太深，先讓客戶進正確類別。
- 高風險要直接導轉。

金融法遵：
- 不可要求敏感資料。
- 不可承諾核准、收益、理賠或處理結果。
- 投資與基金資訊只能提供一般說明。

UX / 文案：
- 不要過度強調 AI。
- CTA 要少而清楚。
- Hero 要說「安全、清楚、必要時轉真人」。

PM：
- MVP 保留服務分類、主要 CTA、高風險提醒。
- 不放複雜報表。
- 不放內部流程。

QA：
- 手機版也要能清楚點擊服務分類。
- 每個按鈕都要有明確下一步。

### 交叉比對結果

保留：
- 開始詢問
- 選擇服務類別
- 需要真人協助
- 高風險提醒
- 8 個服務分類

刪除：
- AI 技術介紹
- 模型能力展示
- 客戶不需要的內部流程
- 任何「保證處理」、「立即核准」類文字

### PM 最終決策

頁面結構：
1. Hero：安全、清楚、必要時有人接手
2. 服務分類
3. 高風險提醒
4. 真人客服入口
5. 常見限制說明

### 最終可執行版本

Hero 主標：
「有問題，先從正確的服務類別開始。」

Hero 副標：
「我們會協助你找到合適的金融服務說明。若涉及疑似詐騙、盜刷、不明交易或需要人工確認的情況，會引導你轉接真人客服。」

服務分類：
- 帳戶與登入
- 信用卡與帳單
- 貸款與進度
- 保單與理賠
- 投資與基金資訊
- 詐騙與不明交易
- 文件上傳與系統問題
- 需要真人客服

高風險提醒：
「若你懷疑遇到詐騙、盜刷、不明交易，請不要提供密碼、驗證碼或完整卡號。建議立即選擇真人客服或撥打官方客服電話。」

CTA：
- 開始詢問
- 選擇服務類別
- 需要真人協助

## 4. 對外 Call Center / 客戶服務頁

### Agent 初版建議

一般客戶：
- 要快速找到服務分類。
- 要看到下一步。
- 不要被要求輸入敏感資訊。

焦慮或生氣客戶：
- 需要明顯的真人客服入口。
- 高風險要有清楚提醒。
- 文案不能冷冰冰。

客服人員：
- 表單不要收太多資料。
- 分類要讓客服接手時知道前因後果。
- 要能降低重複詢問。

金融法遵：
- 不可要求 OTP、密碼、完整卡號。
- 不可承諾結果。
- 投資問題不可變成建議。

UX 設計師：
- 服務分類、問題描述、是否需要真人三步即可。
- 高風險入口要固定可見。
- 無效按鈕刪除。

PM：
- MVP 做「分類、問題描述、高風險、真人轉接」。
- 進階版再做案件追蹤與智能摘要。

QA：
- 驗證每個按鈕都有下一步。
- 驗證高風險能轉真人。
- 驗證不蒐集敏感資料。

### 交叉比對結果

保留欄位：
- 服務類別
- 問題描述
- 是否疑似詐騙 / 盜刷 / 不明交易
- 需要真人協助
- 聯絡方式後續由正式安全流程收集

刪除欄位：
- 完整身分證字號
- 完整卡號
- CVV
- OTP
- 密碼
- 過細的內部案件分類

按鈕：
- 送出問題
- 轉真人客服
- 疑似詐騙 / 盜刷
- 返回服務類別

### PM 最終決策

MVP 流程：
1. 客戶選服務類別
2. 客戶輸入問題描述
3. 系統判斷是否高風險
4. 一般問題給安全說明
5. 高風險轉真人 / 防詐 / 風控

進階版：
- 客戶情緒偵測
- 客服交接摘要
- SLA 追蹤
- 案件編號
- 客服後台同步

### 高風險提醒

「請不要在此輸入密碼、驗證碼、完整卡號或完整身分證字號。若你懷疑遇到詐騙、盜刷或不明交易，請立即選擇真人客服。」
